# 認證準備工作流程指引

本指引涵蓋老虎機認證準備的完整工作流程，包括 GLI-11 與 GLI-19 合規要求、認證文件準備、市場監管資訊、認證時程與費用預估，以及 RTP 門檻警告機制。

---

## 1. GLI-11 合規要求

### 1.1 標準概述

GLI-11（Gaming Laboratories International Technical Standard - Electronic Gaming Devices in Casinos）是全球最廣泛採用的電子博弈機技術標準，適用於實體賭場中的老虎機與電子博弈設備。

**適用範圍**

- 實體賭場中的老虎機（Slot Machines）
- 電子桌面遊戲（Electronic Table Games）
- 視頻彩票終端機（Video Lottery Terminals, VLT）
- 伺服器端博弈系統（Server-Based Gaming）

### 1.2 核心技術要求

#### RNG 要求

| 要求項目 | 說明 | 驗證方式 |
|----------|------|----------|
| 統計隨機性 | RNG 輸出必須通過 NIST SP 800-22 統計測試套件 | 產生至少 100 萬個隨機數進行 15 項測試 |
| 不可預測性 | 無法從先前輸出推導後續輸出 | 原始碼審查確認使用 CSPRNG |
| 不可重現性 | 外部人員無法重現 RNG 序列 | 種子管理機制審查 |
| 獨立性 | 每次旋轉結果獨立於前次結果 | 序列相關性測試與自相關測試 |
| 均勻分佈 | RNG 輸出在指定範圍內均勻分佈 | 卡方檢定（Chi-Square Test） |

#### 遊戲邏輯要求

| 要求項目 | 說明 |
|----------|------|
| 結果確定性 | RNG 輸出一旦產生，遊戲結果必須唯一確定，不可被後續邏輯修改 |
| 規則一致性 | 實際遊戲行為必須與 Paytable 及規則說明完全一致 |
| 獎金準確性 | 所有獎金計算必須精確，不可有捨入誤差導致的系統性偏差 |
| 顯示正確性 | 玩家看到的符號矩陣必須與內部計算結果一致 |

#### 資金管理要求

| 要求項目 | 說明 |
|----------|------|
| 餘額準確性 | 玩家餘額在任何時刻必須準確反映所有交易 |
| 交易原子性 | 投注扣除與獎金發放必須為原子操作，不可出現中間狀態 |
| 斷線恢復 | 旋轉過程中斷線時，必須能正確恢復遊戲狀態 |
| 審計追蹤 | 所有資金變動必須有完整的審計日誌 |

#### 軟體完整性要求

| 要求項目 | 說明 |
|----------|------|
| 程式碼簽章 | 關鍵遊戲邏輯模組必須經過數位簽章驗證 |
| 防篡改機制 | 遊戲軟體必須具備偵測未授權修改的能力 |
| 版本控制 | 每個認證版本必須有唯一的版本識別碼 |

### 1.3 RTP 相關要求

- 理論 RTP 必須在遊戲規則或 Paytable 中明確標示
- 實際 RTP（透過模擬驗證）與理論 RTP 的誤差必須小於 0.5%
- 部分司法管轄區要求最低 RTP（如 Nevada 要求 ≥ 75%，New Jersey 要求 ≥ 83%）

## 2. GLI-19 合規要求

### 2.1 標準概述

GLI-19（Gaming Laboratories International Technical Standard - Interactive Gaming Systems / Remote Gaming Server）適用於遠端遊戲伺服器（RGS）與互動式博弈系統，涵蓋線上老虎機與行動博弈平台。

**適用範圍**

- 遠端遊戲伺服器（Remote Gaming Server, RGS）
- 線上博弈平台（Online Gaming Platforms）
- 行動博弈應用程式（Mobile Gaming Applications）
- 點對點博弈系統（Peer-to-Peer Gaming Systems）

### 2.2 核心技術要求

#### 通訊安全要求

| 要求項目 | 說明 |
|----------|------|
| 傳輸加密 | 所有客戶端與伺服器間的通訊必須使用 TLS 1.2 或以上版本加密 |
| 資料完整性 | 傳輸資料必須具備完整性驗證機制（如 HMAC） |
| 會話管理 | 會話 Token 必須具備過期機制與防重放攻擊保護 |
| 身份驗證 | 玩家身份必須經過多因素驗證或等效安全機制 |

#### 伺服器端 RNG 要求

| 要求項目 | 說明 |
|----------|------|
| 伺服器端執行 | 所有影響遊戲結果的 RNG 運算必須在伺服器端執行 |
| 客戶端隔離 | 客戶端不得影響或預測 RNG 輸出 |
| 多租戶隔離 | 不同玩家的 RNG 實例必須相互隔離 |
| 故障恢復 | RNG 模組故障時必須安全停止遊戲，不可產生偏差結果 |

#### 玩家保護要求

| 要求項目 | 說明 |
|----------|------|
| 自我排除 | 系統必須支援玩家自我排除功能 |
| 存款限制 | 系統必須支援玩家設定投注上限 |
| 會話時間限制 | 系統必須支援遊戲時間提醒機制 |
| 未成年人保護 | 系統必須具備年齡驗證機制 |
| 負責任博弈資訊 | 遊戲介面必須顯示負責任博弈相關資訊與求助連結 |

#### 系統架構要求

| 要求項目 | 說明 |
|----------|------|
| 高可用性 | 系統必須具備故障轉移（Failover）機制 |
| 資料備份 | 遊戲資料與審計日誌必須定期備份 |
| 災難恢復 | 必須制定並測試災難恢復計畫（DRP） |
| 效能監控 | 系統必須具備即時效能監控與告警機制 |

### 2.3 GLI-11 與 GLI-19 差異比較

| 面向 | GLI-11（實體機台） | GLI-19（遠端伺服器） |
|------|-------------------|---------------------|
| 部署環境 | 實體賭場 | 線上 / 行動平台 |
| RNG 位置 | 機台內部 | 伺服器端 |
| 通訊安全 | 內部網路 | 網際網路（需 TLS 加密） |
| 玩家驗證 | 實體場所管控 | 線上身份驗證 |
| 負責任博弈 | 基本要求 | 強化要求（存款限制、自我排除等） |
| 審計日誌 | 本地儲存 | 集中式儲存與備份 |
| 軟體更新 | 現場更新 | 遠端部署（需版本控制） |

## 3. 認證文件準備

認證測試實驗室要求提交完整的技術文件。以下為七項核心認證文件的準備指引。

### 3.1 文件一：RNG 驗證報告

**目的**：證明遊戲使用的隨機數生成器符合密碼學安全標準。

**必要內容**

| 章節 | 內容 |
|------|------|
| 演算法說明 | CSPRNG 演算法名稱、版本、實作來源（如 OpenSSL、Web Crypto API） |
| 種子管理 | 熵值來源、種子長度（建議 ≥ 256 位元）、更新策略 |
| 獨立性保證 | 說明每次旋轉結果如何確保獨立於前次結果 |
| 統計測試結果 | NIST SP 800-22 測試套件的完整測試報告（至少 100 萬個樣本） |
| 原始碼附件 | RNG 模組的完整原始碼（供實驗室審查） |

**準備建議**

- 在提交前先執行 NIST SP 800-22 全部 15 項測試，確保所有 p-value > 0.01
- 附上 Diehard 測試結果作為補充證據
- 明確標示 RNG 模組的程式碼邊界，區分 RNG 核心與遊戲邏輯

### 3.2 文件二：RTP 準確性證明

**目的**：證明遊戲的理論 RTP 計算正確，且與實際模擬結果一致。

**必要內容**

| 章節 | 內容 |
|------|------|
| 理論 RTP 推導 | 完整的數學推導過程，列出所有符號組合及其機率與獎金 |
| 各功能 RTP 貢獻 | 基礎遊戲、Free Spin、Bonus Round、Multiplier、Progressive Jackpot 各自的 RTP 貢獻 |
| 蒙地卡羅模擬 | 至少 1,000 萬次旋轉的模擬結果，含 RTP 收斂曲線 |
| 誤差分析 | 理論值與模擬值的誤差（應 < 0.5%） |
| 試算表附件 | 完整的 RTP 計算試算表（Excel 或等效格式） |

**準備建議**

- 模擬次數建議達到 1 億次以提高統計信心
- 提供不同投注金額下的 RTP 驗證（確認 RTP 不隨投注金額變化）
- 若有多個 RTP 設定（如營運商可調整），需分別提供每個設定的證明

### 3.3 文件三：功能邏輯說明

**目的**：完整描述遊戲的所有功能邏輯，供實驗室驗證遊戲行為的正確性。

**必要內容**

| 章節 | 內容 |
|------|------|
| 遊戲概述 | 捲軸配置（如 5×3）、賠線數量、投注範圍 |
| Spin Lifecycle | 六階段處理流程的詳細說明 |
| 規則引擎 | Wild 替代、Payline 匹配、Scatter 觸發、Multiplier 套用的完整規則 |
| 獎勵功能 | Free Spin、Bonus Round 等所有獎勵功能的觸發條件與運作邏輯 |
| 狀態轉換圖 | 遊戲狀態機的完整轉換圖（基礎遊戲 ↔ Free Spin ↔ Bonus Round） |
| 邊界條件 | 最大獎金上限、同時觸發多個功能時的處理優先順序 |

### 3.4 文件四：Paytable 驗算

**目的**：證明 Paytable 中所有符號組合的獎金倍數與出現機率正確。

**必要內容**

| 章節 | 內容 |
|------|------|
| 符號定義 | 所有符號的名稱、分類（高/中/低賠付、特殊符號） |
| Reel Strip 配置 | 每個捲軸的完整符號序列與 Virtual Reel 權重表 |
| 獎金倍數表 | 每種符號組合（3/4/5 個匹配）的獎金倍數 |
| 機率計算 | 每種獲勝組合的出現機率推導 |
| 交叉驗證 | Paytable 獎金與 RTP 計算的交叉驗證結果 |

### 3.5 文件五：負責任博弈 UX 設計

**目的**：說明遊戲如何實作玩家保護功能，符合負責任博弈要求。

**必要內容**

| 章節 | 內容 |
|------|------|
| 存款限制 | 每日/每週/每月投注上限的 UI 流程與實作說明 |
| 自我排除 | 暫停與永久停止機制的 UI 流程與實作說明 |
| 會話時間提醒 | 遊戲時間提醒的觸發條件、顯示方式與玩家操作選項 |
| 勝負追蹤 | 淨盈虧金額的顯示位置與更新頻率 |
| 風險訊息 | RTP 百分比與博弈風險提示的顯示方式 |
| 自動播放管控 | 自動播放功能的限制條件（依目標市場調整） |
| UI 截圖 | 所有負責任博弈功能的 UI 截圖或線框圖 |

### 3.6 文件六：錯誤處理機制

**目的**：說明遊戲在各種異常情境下的處理方式，確保玩家權益不受損害。

**必要內容**

| 章節 | 內容 |
|------|------|
| 網路斷線處理 | 旋轉過程中斷線的恢復機制（結果保留、餘額回復） |
| 伺服器故障 | 伺服器當機時的遊戲狀態保存與恢復策略 |
| 併發衝突 | 同一帳戶多裝置同時操作的處理方式 |
| 資金異常 | 餘額不一致時的偵測與修正機制 |
| RNG 故障 | RNG 模組異常時的安全停止機制 |
| 錯誤碼定義 | 所有錯誤碼的完整清單與對應的玩家提示訊息 |
| 回退機制 | 交易失敗時的回退（Rollback）流程 |

### 3.7 文件七：審計日誌格式

**目的**：定義審計日誌的格式與欄位，供監管機構審查遊戲公平性。

**必要內容**

| 章節 | 內容 |
|------|------|
| 日誌格式 | JSON 結構定義（參見 rng-game-logic.md 第 6 節） |
| 欄位說明 | 每個欄位的資料類型、必要性與用途說明 |
| 日誌範例 | 至少 3 筆完整的日誌範例（含獲勝、未獲勝、觸發獎勵回合） |
| 儲存策略 | 儲存方式、保留期限、加密機制 |
| 存取控制 | 日誌存取權限定義與存取日誌記錄 |
| 查詢介面 | 提供給監管機構的日誌查詢工具或 API 說明 |

### 3.8 文件準備檢查清單

提交認證前，請確認以下所有文件均已準備完成：

- [ ] RNG 驗證報告（含 NIST SP 800-22 測試結果）
- [ ] RTP 準確性證明（含蒙地卡羅模擬報告）
- [ ] 功能邏輯說明（含狀態轉換圖）
- [ ] Paytable 驗算（含完整機率計算）
- [ ] 負責任博弈 UX 設計（含 UI 截圖）
- [ ] 錯誤處理機制（含錯誤碼清單）
- [ ] 審計日誌格式（含日誌範例）

> ⚠️ 缺少任何一項文件將導致認證審查延遲或退回。建議在提交前由內部品質保證團隊進行完整性審查。

## 4. 市場監管資訊

### 4.1 主要司法管轄區監管機構

| 司法管轄區 | 監管機構 | 適用標準 | 特色 |
|-----------|----------|----------|------|
| **馬爾他** | Malta Gaming Authority (MGA) | GLI-19 + MGA 技術標準 | 歐盟內最受歡迎的博弈牌照，認可度高 |
| **英國** | UK Gambling Commission (UKGC) | GLI-19 + UKGC LCCP/RTS | 全球最嚴格的監管體系之一，強調玩家保護 |
| **安大略省（加拿大）** | Alcohol and Gaming Commission of Ontario (AGCO) / iGaming Ontario | GLI-19 + AGCO 標準 | 北美最大的合法線上博弈市場之一 |
| **直布羅陀** | Gibraltar Gambling Commissioner | GLI-19 + 直布羅陀技術標準 | 歷史悠久的博弈監管區，稅率優惠 |
| **曼島** | Isle of Man Gambling Supervision Commission (GSC) | GLI-19 + GSC 技術標準 | 以靈活的監管框架著稱 |
| **瑞典** | Spelinspektionen (Swedish Gambling Authority) | GLI-19 + 瑞典博弈法 | 2019 年開放線上博弈市場 |
| **內華達州（美國）** | Nevada Gaming Control Board (NGCB) | GLI-11 + Nevada 技術標準 | 全球最知名的實體博弈監管區 |
| **紐澤西州（美國）** | New Jersey Division of Gaming Enforcement (DGE) | GLI-11/GLI-19 + NJ 技術標準 | 美國線上博弈先驅州 |
| **菲律賓** | Philippine Amusement and Gaming Corporation (PAGCOR) | GLI-19 + PAGCOR 標準 | 亞太地區主要的線上博弈牌照發放機構 |
| **庫拉索** | Curaçao Gaming Control Board | 基本技術要求 | 門檻較低，適合初期進入市場 |

### 4.2 各市場特定合規要求

#### 馬爾他（MGA）

| 要求類別 | 具體要求 |
|----------|----------|
| RTP 要求 | 無強制最低 RTP，但須在遊戲規則中明確標示 |
| RNG 認證 | 必須由 MGA 認可的測試實驗室認證 |
| 負責任博弈 | 必須實作存款限制、自我排除、會話時間提醒 |
| 資料保護 | 須符合 GDPR 要求 |
| 審計日誌 | 保留至少 5 年 |
| 自動播放 | 允許，但須提供停止條件設定 |

#### 英國（UKGC）

| 要求類別 | 具體要求 |
|----------|----------|
| RTP 要求 | 須在遊戲介面中明確顯示 RTP 百分比 |
| RNG 認證 | 必須由 UKGC 認可的測試實驗室認證 |
| 負責任博弈 | 最嚴格的要求：強制存款限制、自我排除（GamStop 整合）、淨盈虧顯示 |
| 自動播放 | **2025 年起嚴格限制**：每次自動旋轉間必須顯示淨盈虧、設定損失上限 |
| 速度限制 | 每次旋轉最少間隔 2.5 秒 |
| 審計日誌 | 保留至少 3 年 |
| 年齡驗證 | 強制年齡驗證，禁止 18 歲以下使用 |

#### 安大略省（AGCO）

| 要求類別 | 具體要求 |
|----------|----------|
| RTP 要求 | 須在遊戲資訊中提供 RTP 範圍 |
| RNG 認證 | 必須由 AGCO 認可的測試實驗室認證 |
| 負責任博弈 | 必須實作存款限制、自我排除、遊戲時間提醒 |
| 在地化 | 須提供英語與法語介面 |
| 資料儲存 | 玩家資料須儲存於加拿大境內 |

#### 美國各州（以紐澤西為例）

| 要求類別 | 具體要求 |
|----------|----------|
| 最低 RTP | ≥ 83%（紐澤西）、≥ 75%（內華達） |
| RNG 認證 | 必須由州認可的測試實驗室認證 |
| 地理圍欄 | 必須實作 GPS/IP 地理位置驗證，確保玩家在合法州境內 |
| 年齡驗證 | 必須驗證玩家年滿 21 歲 |
| 合作夥伴 | 線上博弈須與持牌實體賭場合作營運 |

### 4.4 未知市場處理

若開發者的目標市場不在上述清單中，建議：

1. 查詢該國家或地區的博弈監管機構官方網站
2. 確認該市場是否接受 GLI-11 或 GLI-19 標準
3. 聯繫當地認可的測試實驗室了解具體要求
4. 諮詢專業的博弈法律顧問

## 5. 認證時程與費用

### 5.1 認證時程預估

| 遊戲類型 | 預估時程 | 說明 |
|----------|----------|------|
| **標準老虎機** | 2–4 個月 | 5×3 捲軸、標準 Payline、基本獎勵功能（Free Spin） |
| **複雜遊戲** | 4–6 個月 | 多重獎勵功能、Progressive Jackpot、創新遊戲機制 |
| **平台認證** | 6–12 個月 | 完整的 RGS 平台認證（含所有遊戲） |
| **增量更新** | 1–2 個月 | 已認證遊戲的小幅更新或新功能添加 |

**影響時程的因素**

| 因素 | 影響 |
|------|------|
| 文件完整性 | 文件不完整將導致退回修改，延遲 2–4 週 |
| 測試缺陷 | 每次測試失敗需修復後重新提交，延遲 1–3 週 |
| 實驗室排程 | 旺季（年初、年末）排程較滿，可能延遲 2–6 週 |
| 司法管轄區審查 | 部分市場（如 UK）需額外的監管機構審查 |
| 遊戲複雜度 | 創新機制需要更多測試時間 |

### 5.2 認證費用範圍

| 項目 | 費用範圍（USD） | 說明 |
|------|-----------------|------|
| **單一遊戲 / 單一司法管轄區** | $35,000–$140,000 | 依遊戲複雜度與市場要求而異 |
| RNG 認證 | $10,000–$30,000 | 首次 RNG 模組認證 |
| 遊戲數學模型審查 | $8,000–$25,000 | RTP 驗算與 Paytable 審查 |
| 功能測試 | $10,000–$50,000 | 遊戲邏輯與規則驗證 |
| 負責任博弈審查 | $5,000–$15,000 | 玩家保護功能驗證 |
| 平台安全審查 | $15,000–$40,000 | 僅適用於 GLI-19（RGS 平台） |
| 額外市場認證 | $15,000–$60,000 / 每個市場 | 已認證遊戲進入新市場的增量費用 |

**費用節省建議**

1. **一次提交完整文件**：避免因文件不完整導致的來回修改費用
2. **選擇多市場認可的實驗室**：減少重複認證
3. **批量認證**：同時提交多款遊戲可獲得批量折扣
4. **內部預測試**：在提交前充分自測，降低測試失敗的重新提交費用
5. **使用已認證的 RNG 模組**：若 RNG 已通過認證，新遊戲可引用既有認證，節省 RNG 認證費用

### 5.3 認證流程時間線

```
第 1–2 週：準備認證文件（七項核心文件）
    ↓
第 3 週：提交文件至測試實驗室
    ↓
第 4–6 週：實驗室初步審查與文件回饋
    ↓
第 7–8 週：修正文件並重新提交（如有需要）
    ↓
第 9–14 週：正式測試階段（RNG、數學模型、功能、安全）
    ↓
第 15–16 週：測試報告與缺陷修復（如有需要）
    ↓
第 17–18 週：最終審查與認證發放
```

> ⚠️ 以上為理想情境的時間線。實際時程可能因文件品質、測試結果與實驗室排程而有所延長。

## 6. RTP 門檻警告機制

### 6.1 警告規則

> ⚠️ **RTP 門檻警告**：當遊戲的 RTP 設定低於 92% 時，某些市場將禁止該遊戲上線。請務必確認目標市場的最低 RTP 要求。

**警告觸發條件**

- 當 RTP < 92% 時：觸發市場限制警告
- 當 RTP ≥ 92% 時：不觸發警告

### 6.2 各市場 RTP 門檻

| 市場 | 最低 RTP 要求 | 說明 |
|------|-------------|------|
| 英國（UKGC） | 無強制最低值，但須明確標示 | 實務上低於 85% 的遊戲難以通過審查 |
| 馬爾他（MGA） | 無強制最低值 | 須在遊戲規則中標示 |
| 內華達州 | ≥ 75% | 美國最低門檻之一 |
| 紐澤西州 | ≥ 83% | 線上與實體均適用 |
| 安大略省 | 無強制最低值 | 須在遊戲資訊中提供 |
| 丹麥 | ≥ 90% | 歐洲較高門檻市場之一 |
| 比利時 | ≥ 84% | 適用於所有博弈類型 |

### 6.3 低 RTP 的市場影響

當 RTP 低於 92% 時，開發者應注意以下限制：

1. **市場進入受限**：部分歐洲市場（如丹麥）要求 RTP ≥ 90%，低於 92% 的遊戲可能無法進入多個市場
2. **玩家接受度降低**：業界標準 RTP 為 94%–98%，低於 92% 的遊戲在玩家評價中通常較差
3. **營運商合作困難**：多數線上博弈營運商偏好 RTP ≥ 94% 的遊戲，低 RTP 遊戲可能難以獲得營運商合作
4. **認證審查加嚴**：部分測試實驗室會對低 RTP 遊戲進行額外審查，確認 RTP 標示清晰且玩家知情

### 6.4 建議行動

若遊戲 RTP 低於 92%：

1. **重新檢視數學模型**：調整符號權重或獎金倍數以提高 RTP
2. **確認目標市場合規**：逐一確認每個目標市場的最低 RTP 要求
3. **明確標示 RTP**：在遊戲介面與規則說明中清楚標示 RTP 百分比
4. **提供 RTP 調整選項**：考慮提供多個 RTP 設定（如 92%、94%、96%），讓營運商依市場需求選擇
5. **諮詢法律顧問**：確認低 RTP 設定在目標市場的合法性
